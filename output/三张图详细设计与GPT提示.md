## Agent 选择说明

本任务基于已有综述正文、论文大纲和图表规划，不是重新定题或重写全文。因此采用 `academic-research-suite / academic-paper / visualization_agent` 作为主 agent，辅以 `structure_architect_agent` 和 `argument_builder_agent` 的思路进行结构对齐：三张图分别服务于“核心贡献框架”“技术实现链路”“可信部署闭环”，并与正文第 1、3、5、6、7 节形成呼应。

## 总体绘图规范

- 图形风格：中文学术论文黑白矢量框图，适配《计算机学报》类双栏排版。
- 页面比例：建议 `figure*` 双栏宽度，横向 16:9 或 4:3；正式制图优先使用 TikZ、draw.io、Visio、PowerPoint 矢量导出 PDF。
- 视觉语言：白底、黑/深灰线条、浅灰填充，不依赖颜色表达核心语义；不同类别用线型、边框粗细、编号、阴影深浅区分。
- 字体层级：图题不放在图内；图内中文标签尽量 4-8 字，辅助说明不超过 12 字；避免长句。
- 统一符号：实线箭头表示主流程，虚线箭头表示反馈/约束，双线框表示高风险或强约束模块，圆角矩形表示功能模块，菱形表示校验门。
- 三图关系：图1回答“本文如何分析”，图2回答“MLLM 如何从输入走到业务输出”，图3回答“如何可信部署并闭环治理”。

## 图2 面向智能电网的 MLLM 技术链路 / Pipeline

### 1. 图的定位

- 放置位置：第 3.3 节之后或第 5 节“知识增强、RAG 与电力领域可信推理”之前。

  - 第 2 节：视觉语言模型、指令微调、RAG、知识图谱、工具调用。
  - 第 3 节：多模态数据类型与任务体系。
  - 第 5 节：知识增强与可信推理。

### 2. 推荐版式

采用横向流水线图，分为 6 个阶段，每个阶段用竖向泳道或大模块表示。

主流程：
`多源输入 -> 数据治理与对齐 -> 多模态表征 -> MLLM 推理 -> 知识/工具增强 -> 业务输出`

底部增加一条虚线反馈链：
`人工反馈/审计日志 -> 数据集更新 -> 知识库更新 -> 模型适配`

### 3. 图内元素

#### 阶段一：多源输入

标题：`多源输入`

模块：
- `巡检图像/视频`
- `红外/传感`
- `时序量测`
- `拓扑/接线图`
- `规程/工单/台账`

输出到下一阶段的箭头标注：`采集与接入`

#### 阶段二：数据治理与对齐

标题：`治理与对齐`

模块：
- `质量检查`
- `脱敏/权限`
- `时空对齐`
- `实体链接`

- `上下文推理`
- `候选生成`
- `不确定性提示`

#### 阶段五：知识/工具增强

标题：`知识与工具增强`

模块：
- `RAG 检索`
- `知识图谱`
- `规程规则库`
- `潮流/仿真/优化工具`
- `安全校验器`

与阶段四之间使用双向箭头，表示模型可以调用外部知识和工具，外部工具也返回校验结果。

#### 阶段六：业务输出

标题：`业务输出`

模块：
- `缺陷说明`
- `诊断证据链`
- `调度建议说明`
- `应急脚本`
- `报告/工单草案`

1. `人工采纳/驳回`
2. `审计日志`
3. `错误样本回流`
4. `知识库更新`
5. `模型持续适配`

注意：反馈链用虚线，不要和主流程混淆。

`MLLM 辅助决策模块`

- `设备限值检查`
- `权限与责任确认`
- `人工复核`

箭头说明：

### 4. 闭环逻辑

主闭环：
`任务触发 -> 风险分级 -> 证据接入 -> MLLM 生成候选建议 -> 强制校验门 -> 人工确认 -> 业务执行/归档 -> 审计反馈 -> 知识库/模型更新`

关键表达：

- MLLM 的输出是“候选建议”，不是自动控制命令。
- 可信部署依赖规则校验、物理仿真、权限审查和人在回路。
- 审计日志和人工反馈用于持续改进知识库、提示模板、评测集和模型适配。
